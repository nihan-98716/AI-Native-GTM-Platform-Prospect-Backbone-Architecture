# AI Agents & Orchestration

## Agent Inventory

The Prospecting workflow is powered by six specialized agents/nodes:

1.  **ProspectResearchAgent:** Sources and ranks target accounts based on ICP definitions.
2.  **ContactEnrichmentAgent:** Identifies and enriches key personas within target accounts.
3.  **IntentSignalAgent:** Analyzes intent signals and recent activities to prioritize outreach.
4.  **ValueHypothesisAgent:** Generates account-specific value propositions grounded in research.
5.  **ApprovalGate:** A human-in-the-loop checkpoint for reviewing hypotheses and drafts.
6.  **OutreachAgent:** Produces personalized outreach messages (email/LinkedIn) for target contacts.

## Tools & Capabilities

Agents interact with the system through a unified `ProspectAgentTools` interface:

- `get_icp`: Loads tenant ICP criteria.
- `get_accounts`: Retrieves tenant-owned account records.
- `search_provider_accounts`: Sourced new accounts from Apollo.
- `get_contacts`: Retrieves enriched contact records.
- `enrich_provider_contacts`: Invokes Apollo for contact discovery.
- `get_signals`: Loads intent data for specific accounts.

## Orchestration & State

### LangGraph Workflow

The orchestration uses **LangGraph** to manage a directed graph of agent nodes.

- **State Passing:** A global `ProspectWorkflowState` object is passed between nodes, accumulating research, contacts, and drafts.
- **Resumability:** Workflow state is persisted in PostgreSQL, allowing long-running agent runs to survive worker restarts or wait for human approval.

### Failure Handling & Retries

- **Retry Policy:** Each agent has a configured `AgentRetryPolicy` (e.g., 2 attempts with exponential backoff).
- **Fallbacks:** Critical agents include fallback logic to return partial results if the LLM provider is unavailable.
- **Graceful Degradation:** The workflow can continue with partial data if enrichment or signal discovery fails for a subset of accounts.

## Human-in-the-Loop

The **ApprovalGate** node pauses the workflow when a human review is required.

- **Trigger:** Configurable per workflow (e.g., `require_human_approval: true`).
- **Interaction:** The workflow status transitions to `waiting_for_approval`. A user reviews the generated hypotheses and outreach drafts in the UI.
- **Resumption:** Upon approval, the workflow resumes at the next node (`OutreachAgent` or completion).

## Traceability

Every agent execution produces a durable **Trace**:

- **Decision Summary:** A high-level rationale for the agent's actions.
- **Tool Invocations:** Detailed logs of every tool call made by the agent.
- **State Transitions:** Snapshots of the workflow state before and after the node run.
- **Inputs/Outputs:** Full structured payloads used and generated by the LLM.

Traces are persisted in the `workflow_runs` and `tool_calls` tables, providing full auditability for every AI decision.

## Prompt Management

Prompts are stored as system templates in `backbone/app/agents/prompts/`:

- `prospect_research_system.txt`
- `contact_enrichment_system.txt`
- `intent_signal_system.txt`
- `value_hypothesis_system.txt`
- `outreach_system.txt`

## Cost & Latency Estimates

The platform includes a costing service that estimates LLM usage based on model-specific token rates.

- **Tracking:** Each agent run records `token_input`, `token_output`, and `estimated_cost`.
- **Latency:** Latency is tracked per node and per tool call to identify performance bottlenecks.
- **Current Models:** Defaulting to `gpt-4.1-mini` for research tasks and `gpt-4.1` for creative generation.
